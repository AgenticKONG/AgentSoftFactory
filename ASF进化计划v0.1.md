## 【1】基础测试项目库建设与分级测试 🏆 【优先级：极高】

### 1.1 基础测试项目库建设 (The Testbed Matrix)
**建设目标：** 建立以 DTHBH 为核心素材，覆盖数据、后端、前端、可视化、叙事五大分支的 L1-L5 全谱系演练场。

#### 📊 核心度量矩阵 (Capability-Progression Grid)
| 分支 (C) | L1: 原子测试 | L2: 模块集成 | L3: 复杂逻辑 | L4: 系统闭环 | L5: 全局叙事 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **DATA (数据)** | 简单正则/清洗 | 多源数据对齐 | 自动数据清洗流 | 知识图谱初步构建 | 语义化人文数据库 |
| **BACK (后端)** | 单一 API 接口 | 逻辑鉴权接口 | 分层架构 API | 外部服务集成 | 自主运维与自愈 |
| **FRONT (前端)**| 静态列表渲染 | 响应式组件 | 状态管理流 | 动态主题换肤 | 沉浸式交互框架 |
| **VIZ (可视化)** | 基础柱状图 | 动态时间轴 | **LifeRiver 组件** | 社交网络图谱 | 风格化叙事呈现 |
| **NARR (叙事)** | 摘要生成 | 专题大纲 | 内容与知识映射 | 智能研究员报告 | DTHBH 全量策展 |

#### 📏 测算点与成功准则 (The Metrics)
- **Primary Metric:** 一次性交付成功率 (1-Shot Pass Rate)。
- **Process Metric:** 达到 QA Pass 所需的平均迭代轮数 (Avg. Iteration Count)。
- **Efficiency Metric:** 任务完成的总 Token 消耗与时长 (Token/Time Cost)。
- **Visual Metric:** 针对 VIZ/FRONT，测算 DEV 对 VD Spec 的还原精度。

#### 🚦 模型切换门禁 (Model Gating)
- **L1-L2 (基础期):** 100% 采用本地模型 (Llama3.2)，用于打磨 P (流程) 和 T (DNA Prompt)。
- **L3 (转折期):** 引入 GLM4.7 / MiniMax-M2.5 担任 PM 角色，测试“强脑带弱手”效率。
- **L4-L5 (成熟期):** 全线引入 Claude-Code / Opus，利用成熟流程实现高价值交付。

### 1.2 主控中心 (Infra Center) 优化
现阶段，基础测试项目库建设、项目运行、测试执行和流程改进的职能都在基建中心（Infra Center，IC），由Infra Agent（IA）负责统筹工作，向Eric汇报。

**IA (Infra Agent) 核心职责：**
1.  **体系维护：** 维护 `PROJECT_MATRIX.md` 度量体系。
2.  **执行监督：** 配置、触发和监督测试项目执行过程（可配置“Project日志存储”和“Daemon Agents抽样”等方式具体执行）。
3.  **异常调度：** 实时处理流程阻塞（Soul Beacon 机制）。
4.  **数据汇报：** 测试数据的整理和分析工作，定期向Eric汇报，提出流程改进建议，（确认后）执行流程改进。
5.  **基建进化：** 基础设施的自我进化（重大决策需向Eric确认）。

**Daemon Agents：**
- 直接接受IA的垂直领导，由IA分配任务并触发。

### 1.3 分级测试执行
- **测试点设计：** 逻辑一致性、Token 消耗、迭代成功率、代码健壮性。
- **演进路径：** L1 (一次过) -> L3 (循环纠错) -> L5 (多 Agent 协同)。
- **执行计划：** 先在本地环境跑通 L1-L2，达标后再开启商业大模型库。
- **异常处理流程：** 监测到 `STALLED` 状态自动触发 Soul Beacon 给大脑。
- **过程改进流程：** 数据采样 -> 分析瓶颈 -> 优化 DNA Prompt -> 验证闭环。

## 【2】ASF 基座改造（去 CoPaw 化） 🏗️

### 2.0 ASF 流程优化与 Project 模块改造 【优先级：高】
**思路：** 完成 ASF 流程的大规模测试和优化，以实现“实战级”的多 Agents 协同项目（如 DTHBH）。坚持“度量-总结-改进”的闭环。

#### 🟢 运行前：配置即契约
- **流程：** Registry（New Project） -> Configure -> Plan(预留) -> Launch.
- **项目配置：** 基本信息 (L等级 / C类型)、T团队 (配置Agents)、P流程 (预设/编辑)、F特性 (目标)、R需求 (Prompt/MD)。

#### 🟡 运行中：透明监控
- **监控：** `WorkStation` 实时流。
- **度量：** 数据驱动决策，形成制度化度量体系，减少人和 TOKEN 的消耗。
- **异常处理：** 能及时发现异常，以预案方式处理，重大问题及时止损并报告。


#### 🔴 运行后：归档与蒸馏
- **工作：** 交付（运维需求）、总结（数据驱动的过程改进）、归档（解散团队，分类存储）。

### 2.1 Agent 个性化配置与 Agents Market 【优先级：中】
**思路：** Agent 实现 Model + Skills + MCPs 的灵活组合，进入 Agents Market (Factory)。

#### 🛠️ 核心架构
- **Hire 机制：** Agent 克隆体以 Hire 方式执行任务，在配置页面仍可编辑。
- **专才专用：** Agent 拥有独立 SOUL 和 MEMORY，但限制其 Skills 数量，维护记忆质量。
- **能力反哺：** 表现优秀的克隆体可将“习得”的技能装载回母体。

#### 📋 模块分解
- **2.1.1 Agents Market (Factory) ：** 展示 Agent技能与历史战绩，详情跳转Agent。
- **2.1.2 Agent：** 配置 Model、Skills、Channels。
- **2.1.3 Models：** 从`Settings`页迁移，动态选择已注册的 Providers。
- **2.1.4 Skills：** 原 `Skills`，保留。
- **2.1.5 MCPs：** 原`MCP`，保留。
- **2.1.6 Channels：**从`Control\Channel`迁移，Agent 克隆体可配备 0-2个外部工具（飞书/钉钉），用于关键节点汇报。
